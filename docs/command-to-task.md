# Command → Task 변환 규칙 (초안)

`jarvis-core`에서 사용자 자연어 명령을 **바로 실행하지 않고**, 먼저 `Task`로 변환해 기록하기 위한 최소 규칙을 정의한다.

## 1) 목적과 범위
- 목적: 자연어 명령을 일관된 형식의 Task로 바꿔 추적 가능성을 확보한다.
- 범위: 이번 단계는 **규칙 정의와 수동 적용 기준**만 다룬다.
- 비범위:
  - Discord 연동
  - 웹/API 서버 구현
  - NLP 모델/자동 분류기 구현
  - 코드 자동 생성 실행

## 2) 입력 명령 해석 규칙

### 2.1 기본 원칙
- 명령문에서 핵심 동사(만들어, 개선해, 추가해 등)와 대상(웹 UI, 보고 시스템, 디스코드 연결 등)을 분리한다.
- 하나의 명령에 여러 목표가 섞이면 **Task를 분리**한다.
- 애매하거나 불명확하면 실행하지 않고, `TODO` Task로 기록 후 추가 확인이 필요함을 `summary`에 남긴다.

### 2.2 명령 패턴(초기)
- 생성/구축 계열: `만들어줘`, `구축해`, `작성해`
- 개선/수정 계열: `개선해`, `정리해`, `보완해`
- 연동/추가 계열: `추가해`, `연결해`, `붙여줘`

해석 방법:
1. 동사 계열로 작업 유형을 고른다.
2. 명사 구(대상 시스템/기능)로 작업 대상을 정한다.
3. 불명확한 제약(범위, 우선순위, 완료 기준)은 `summary`의 확인 필요 항목으로 남긴다.

## 3) Task 필드 생성 규칙

기본 필드는 `docs/task-model.md`를 따른다.

### 3.1 `title` 생성 규칙
- 형식: `<action> <target>`
- 규칙:
  - 영어 소문자 기반의 짧은 작업 제목 사용
  - 구현 방법보다 목표 중심으로 작성
- 예:
  - `build web ui`
  - `improve reporting system`
  - `add discord integration`

### 3.2 `summary` 생성 규칙
- 1~3문장으로 아래를 포함한다.
  1. 무엇을 하려는지(목표)
  2. 이번 단계에서 다루는 범위
  3. 미확정/확인 필요 사항(있다면)
- 구현 세부 코드 계획보다 운영 관점 요약을 우선한다.

### 3.3 `repo` 선택 기준
- 원칙: **구조/정책/기록** 중심 작업은 `jarvis-core`.
- 실제 제품 기능 구현(런타임 코드, 서비스 로직)은 해당 서브 저장소.
- 판단이 애매하면 기본값은 `jarvis-core`로 기록하고, `summary`에 repo 재확인 필요를 명시한다.

### 3.4 초기 상태
- 새 Task의 기본 상태는 항상 `TODO`.
- 바로 실행 가능한 명령이어도 선기록 후진행 원칙을 유지한다.

### 3.5 `id` 생성 방식
- 형식: `task-####-slug`
- 절차:
  1. `memory/tasks/`에 있는 기존 Task 파일에서 가장 큰 번호(`####`)를 찾는다.
  2. 다음 번호를 4자리 0패딩으로 할당한다.
  3. 대상 중심 slug를 소문자-하이픈으로 붙인다.
- 예:
  - 기존 최대가 `task-0002-*`이면 다음은 `task-0003-...`

## 4) 명령 → Task 변환 예시

### 예시 1
입력:
- `웹 UI 만들어줘`

출력:
- `id`: `task-0001-web-ui`
- `title`: `build web ui`
- `status`: `TODO`
- `repo`: `subrepo-web` (실제 UI 구현 저장소가 정해져 있다면)
- `summary`: `사용자용 웹 UI 구축 작업을 정의한다. 이번 Task는 UI 범위와 화면 구성 기준 정리에 초점을 둔다. 기술 스택과 상세 요구사항 확인이 필요하다.`

### 예시 2
입력:
- `보고 시스템 개선해`

출력:
- `id`: `task-0003-report-improvement`
- `title`: `improve reporting system`
- `status`: `TODO`
- `repo`: `jarvis-core`
- `summary`: `보고 체계의 구조와 문서 흐름을 개선한다. 이번 Task는 보고 템플릿/상태 흐름/기록 기준 정리를 포함한다. 자동화 구현은 범위에서 제외한다.`

### 예시 3
입력:
- `디스코드 연결 추가해`

출력:
- `id`: `task-0004-discord-integration`
- `title`: `add discord integration`
- `status`: `TODO`
- `repo`: `subrepo-integration` (실제 연동 구현 저장소가 있다면)
- `summary`: `디스코드 연동 작업을 Task로 등록한다. 이번 단계는 연동 요구사항과 인터페이스 정의까지만 포함한다. 토큰/권한/배포 방식은 추가 확인이 필요하다.`

## 5) Task 생성 절차 (수동 운영)
1. 명령 입력 수신
2. 명령을 작업 단위로 분해(필요 시 다중 Task 분리)
3. `title`, `summary`, `repo`, `status=TODO`, `id` 규칙에 따라 Task 메타데이터 작성
4. `memory/tasks/<task-id>.md` 파일 생성
5. 초기 상태/확인 필요 사항 기록
6. 검토 후 다음 실행 Task를 선택

## 6) 금지 규칙
- 애매한 명령을 즉시 실행하지 않는다.
- 실행보다 먼저 Task로 기록한다.
- 여러 작업이 섞인 명령은 하나의 Task로 뭉치지 않는다.
- Task 생성 단계에서 코드 구현/배포/외부 연동을 진행하지 않는다.

## 7) 운영 메모
- 이 문서는 초기 초안이며, 실제 Task 누적 후 변환 예시와 slug 사전을 보강한다.
- 자동 변환기는 추후 단계에서 검토하며, 현재는 사람이 읽고 적용 가능한 규칙 유지가 우선이다.
